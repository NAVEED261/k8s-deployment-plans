Project 2 — K8s Planning

Kubernetes Deployment Plans for AI-Native Apps

Two detailed infrastructure plans, zero code. The whole point was to think through how you'd actually run these systems — who can access what, what happens when a pod crashes, how secrets rotate.

View on GitHub Plan 1 — Task Manager Plan 2 — OpenClaw
Two scenarios
Scenario 01
AI Native Task Manager
UI, backend APIs, a task agent, and a notification service. Four namespaces, six secrets, RBAC scoped so the agent reads exactly what it needs — nothing more.
4 namespaces 6 deployments 2 statefulsets HPA
Scenario 02
AI Employee — OpenClaw
Personal AI worker with email, calendar, and code execution access. gVisor sandbox, default-deny networking, Vault-backed secrets, immutable audit logs.
5 namespaces gVisor Vault Zero Trust
What each plan covers
Namespaces for logical isolation
Deployments and StatefulSets
ClusterIP, NodePort, LoadBalancer
CPU and memory sizing per service
ConfigMaps for environment config
Secrets with rotation schedules
RBAC roles and service accounts
Inter-service DNS and NetworkPolicy

The hardest part was the OpenClaw plan. When an AI agent has access to someone's inbox and can run code, the security questions get real fast. Default-deny networking, a sandboxed execution namespace, audit logs for every action. Most K8s tutorials skip all of this.

Reusable skill
K8 Planning Skill
A reusable agent skill for any future project. Give it a list of services, a security level, and expected traffic — it walks through namespace design, workload classification, service types, resource sizing, and a full security checklist. The K8P framework makes the process repeatable without starting from scratch every time.
Includes Anthropic Skill Creator eval — 3 test cases
